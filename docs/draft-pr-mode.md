# Draft PR Mode

## What it means

In `draft-pr` mode the agent behaves exactly like patch-only/branch-only: it may
read and edit files and run tests, but must **not** commit, push, or use `gh`. The
harness runs the agent in an isolated git worktree. When the run completes, a
separate operator step delivers the changes as a draft PR via a new script:
`./harness/create-draft-pr.sh`.

The script stages all uncommitted changes, creates a branch named
`agent-lab/<task_id>`, commits with a descriptive message, and opens a draft PR
on GitHub — but only when you explicitly approve via confirmation flags.

## When to use draft-pr

| Use draft-pr when… | Use branch-only when… |
|---|---|
| You want changes delivered as a draft PR, not a local branch | You want to review locally before pushing anywhere |
| The repo process requires draft PRs from bots | The change is exploratory and may not be merged |
| You want the PR body auto-generated from the run report | You plan to write your own PR description |
| You want safety gates before push and PR creation | You prefer full manual control over git operations |

See [patch-only-mode.md](patch-only-mode.md) and [branch-only-mode.md](branch-only-mode.md)
for alternative workflows.

## Running a draft-pr task

```bash
# 1. Create a run directory from your task file
run_dir=$(./harness/create-run.sh examples/tasks/my-task.yaml)

# 2. Clone the repo and create the worktree
./harness/prepare-repo.sh "$run_dir"

# 3. Run the agent
./harness/run-claude.sh "$run_dir"

# 4. (Optional) Verify outputs
./harness/verify-run.sh "$run_dir"

# 5. Deliver as a draft PR (with confirmation gates)
./harness/create-draft-pr.sh "$run_dir" --confirm-push --confirm-pr
```

The `create-draft-pr.sh` script creates a branch named `agent-lab/<task_id>` inside
the worktree, stages and commits with:

```
feat(agent-lab): <title>

Run ID: <run_id>
Mode: draft-pr
Task: <task_id>

Generated by TSD Agent Lab harness. Human review required before push.
```

Then pushes to `origin/<branch>` and opens a draft PR.

## The confirmation gates

`create-draft-pr.sh` **always** prints a preview first, showing:
- Target repo
- Branch name
- Base ref (default: `main`)
- PR title (from task)
- PR body (generated from run report)

### Without --confirm-push

```bash
./harness/create-draft-pr.sh "$run_dir"
```

The script stops after printing the preview and displays the push command. **Nothing is
pushed or merged.** Use this to review the diff locally before committing.

### With --confirm-push

```bash
./harness/create-draft-pr.sh "$run_dir" --confirm-push
```

Runs `git push -u origin <branch>` from the worktree. The branch is now on GitHub,
but the PR is **not** created yet. Use this when you are satisfied with the diff
and ready to make the branch visible.

### With --confirm-pr (implies --confirm-push)

```bash
./harness/create-draft-pr.sh "$run_dir" --confirm-push --confirm-pr
```

After pushing, runs:
- `gh auth status` (fails safely with guidance if you are not authenticated to GitHub)
- `gh pr create --draft --base <base> --head <branch> --title <title> --body <body>`

The PR is opened as a **draft** with no reviewers, labels, or assignees. The PR
body is auto-generated from `summary.md` in the run directory, falling back to
run metadata and `changed-files.txt` if the summary is missing. All `@mentions`
in the body are backtick-wrapped (e.g., `@bot-name`) so no other users or bots
are pinged.

## Example: broken-build flow

Use `lab-action draft-fix` to fix a broken build, review the diff, then deliver
as a draft PR:

```bash
lab-action draft-fix INC-0001              # bot fixes broken build -> patch + report
#   ... operator reviews the diff in the worktree ...
./harness/create-draft-pr.sh <run-dir> --confirm-push --confirm-pr   # deliver as draft PR
```

The operator has full visibility into what the agent changed before any push
or PR occurs.

## Dry-run

```bash
./harness/create-draft-pr.sh "$run_dir" --confirm-push --confirm-pr --dry-run
```

Prints every intended git and gh command — `git push`, `gh auth status`, `gh pr create` —
without making any changes. Use this to verify the script's behavior before running
for real.

## The generated PR

- **Draft status:** PR is created as a draft (not ready for merge).
- **Body:** Auto-generated from `summary.md` in the run report. Falls back to run
  metadata and `changed-files.txt` if summary is missing.
- **Mentions neutralized:** All `@mentions` are backtick-wrapped (e.g., `@user`
  becomes `` `@user` ``) so no notifications are sent.
- **No reviewers, labels, or assignees:** The PR is clean and awaits manual review.

## Safety emphasis

**A human must review the local diff FIRST.** The script only runs when you
explicitly invoke it with the `--confirm-push` and `--confirm-pr` flags.

Use draft-pr mode only in repos whose owners' process allows draft PRs from the
bot. This mode is **never** wired to GitHub comments or issue events; comment
triggering `/agent fix` is an explicit non-goal. Only `/agent review` is
comment-triggerable.

## Failure modes

**Wrong mode.** The script checks `mode == draft-pr` in `run-metadata.json` and
exits with an error if the mode is different (e.g., `branch-only` or `patch-only`).

**No changes found.** The script exits cleanly with a warning. Possible causes:
the agent made no edits, or it committed changes already (check `git log` in
the worktree).

**Branch already exists.** Use `--force` to overwrite:

```bash
./harness/create-draft-pr.sh "$run_dir" --confirm-push --confirm-pr --force
```

Or delete the branch manually: `git branch -D agent-lab/<task_id>`.

**GitHub not authenticated.** If `gh auth status` fails, the script stops and
prints guidance for logging in. Run `gh auth login` and retry.

**Worktree missing.** `run-metadata.json` contains `worktree_path`. If that
directory no longer exists, re-run `prepare-repo.sh` to recreate it.
